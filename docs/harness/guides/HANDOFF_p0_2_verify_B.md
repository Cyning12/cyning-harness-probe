## 3. 待执行步骤（按顺序）

### 3.1 修改 `tests/test_verify.py` 的 `_make_task`

新增一个可选参数 `create_test_stub: bool = False`。当为 `True` 时，在 `tmp_path` 下创建 `tests/test_<slug>.py`，并确保 `tests/__init__.py` 存在。

`slug` 由 `_expected_test_files` 的启发式规则决定：对于 `task_test_verify_v1.md`，去掉前缀 `task_` 和版本后缀 `_v1`，得到 `test_verify`，所以临时测试文件应为 `tests/test_test_verify.py`（注意：是 `_expected_test_files` 的 glob 模式 `test_test_verify*.py`）。

更稳妥的做法是：让任务单文件名映射到一个唯一的、临时测试文件，例如 `task_verify_cli_<test_name>_v1.md`，并在 `tmp_path/tests/test_verify_cli_<test_name>.py` 创建对应测试文件。需要确认 `_expected_test_files` 的 glob 能匹配上。

推荐映射：

- 任务单文件：`tmp_path/task_verify_cli_markdown_v1.md`
- 测试文件：`tmp_path/tests/test_verify_cli_markdown.py`

这样 `_expected_test_files` 返回 `[tmp_path/tests/test_verify_cli_markdown.py]`，CLI 在该临时目录下运行 `pytest` 时只跑这个文件，不会递归。

### 3.2 修改 `_make_task` 内部实现

```python
def _make_task(
    tmp_path: Path,
    *,
    graph_delta: str = "none",
    human_gates: list[dict[str, Any]] | None = None,
    test_strategy: str = "not_applicable",
    slug: str = "test_verify",
    create_test_stub: bool = False,
) -> Path:
    gates = human_gates or [...]
    frontmatter = f"""---
id: test-task-v1
title: Test Task
status: draft
module_id: VERIFY
graph_delta: {graph_delta}
test_strategy: {test_strategy}
freeze_id: v0.10.0
human_gates:
"""
    # ... existing gate loop ...
    p = tmp_path / f"task_{slug}_v1.md"
    p.write_text(frontmatter, encoding="utf-8")

    if create_test_stub:
        tests_dir = tmp_path / "tests"
        tests_dir.mkdir(exist_ok=True)
        (tests_dir / "__init__.py").write_text("", encoding="utf-8")
        stub = tests_dir / f"test_{slug}.py"
        stub.write_text(
            '"""Stub test generated by test_verify.py."""\n\n'
            'def test_stub():\n'
            '    assert True\n',
            encoding="utf-8",
        )
    return p
```

### 3.3 修改 `TestVerifyCLI` 用例

每个 CLI 用例传入不同的 `slug` 并启用 `create_test_stub=True`：

- `test_cli_verify_task_markdown` → `slug="verify_cli_markdown"`
- `test_cli_verify_task_json` → `slug="verify_cli_json"`
- `test_cli_verify_ci_returns_nonzero_on_failure` → `slug="verify_cli_ci"`
- `test_cli_verify_dir_batch` → 创建两个任务单，分别用 `slug="verify_cli_dir_a"` 和 `slug="verify_cli_dir_b"`
- `test_cli_verify_dir_empty` → 无需修改，保持空目录
- `test_cli_pre_spawn_verify` → `slug="verify_cli_pre_spawn"`

同时需要确保 CLI 子进程在 `tmp_path` 下运行，而不是 `REPO_ROOT`，因为临时测试文件在 `tmp_path/tests/` 下。但当前 CLI 的 `cmd_verify` 中 `repo_root` 是通过 `_repo_root()` 从 `verify.py` 获取的，会固定指向仓库根目录，而不是任务单所在目录。

**这是 B 方案的关键细节**：需要确认 `cmd_verify` 是否支持指定 `--graph-root` 或在任务单所在目录运行外部检查。当前 CLI 的 `verify_task` 调用没有传递 `graph_root`，但 `pytest` 的 cwd 是 `_repo_root()`，所以即使临时测试文件在 `tmp_path/tests/` 下，pytest 也找不到。

### 3.4 解决 pytest cwd 问题（关键）

有两种选择：

**B-1. 让 `_run_subprocess_check` 的 cwd 改为任务单所在目录**

在 `verify_task` 中，外部检查使用 `task_path_resolved.parent` 作为 cwd，而不是 `repo_root`。这更符合直觉：任务单和测试文件放在同一目录下，外部检查就在该目录运行。

修改 `verify_task`：

```python
    # 外部检查在任务单所在目录运行，允许 CLI 测试使用临时测试文件
    check_cwd = task_path_resolved.parent
```

然后外部检查（pytest/ruff/mypy）使用 `check_cwd` 作为 cwd。

**B-2. 在 CLI 参数里增加 `--check-cwd` 或让 `--dir` 模式使用 dir 作为 cwd**

更复杂，但保留了仓库根目录运行的默认行为。用户选择 B 方案，B-1 更简单直接。

**推荐采用 B-1**：将外部检查 cwd 改为 `task_path_resolved.parent`，并同步修改单元测试 `test_external_checks_injected` 的断言，因为 `pytest_check.message` 中不再包含 `tests/test_verify.py`（因为测试文件在临时目录下）。

### 3.5 调整 `test_external_checks_injected` 断言

如果采用 B-1，原来的断言 `assert "tests/test_verify.py" in pytest_check.message` 不再成立。应改为：

```python
pytest_check = [c for c in report.checks if c.name == "pytest"][0]
assert "test_" in pytest_check.message
assert "pytest" in pytest_check.message
```

或确认 `pytest_check.message` 包含任务单对应的测试文件路径。

## 4. 已知陷阱与注意事项

1. **递归 pytest**  
   任何让 `verify_task` 运行 `tests/test_verify.py` 的路径都会触发递归。B 方案通过临时测试文件和任务单所在目录 cwd 避免这一点。

2. **`completed` 状态 normalize**  
   Pydantic 的 `HumanGateStatus` 会把 `completed` 自动映射为 `approved`。`verify.py` 中的 `_extract_raw_gate_statuses` 通过原始 frontmatter 文本解析，strict 模式才能识别 `completed` 为非法。不要依赖 `gate.status == HumanGateStatus.completed`。

3. **`_expected_test_files` 的 slug 提取**  
   当前逻辑是：去掉前缀 `task_`，去掉末尾版本号（如 `_v1` / `_v0_1_0_v1`），剩余部分作为 slug。测试文件 glob 模式为 `test_<slug>*.py`。因此临时测试文件命名需要与任务单文件名一致。

4. **不要删除 `docs/_tech_graph/89_verify.graph.yaml` 或 CHANGELOG 条目**  
   这些已经完成，除非有功能变更才需同步更新。

5. **CLI 子进程超时**  
   `TestVerifyCLI` 中 `timeout=60` 在递归情况下会超时。修复后 60 秒足够，但如果 CI 机器慢，可以适当放宽到 120 秒。
